  # Topo Tracer Overview

## What is Topo Tracer?

Topo Tracer is a log-chain-first tracing logger.

The core idea is simple:

```txt
log -> log -> log -> log
```

Every trace is first stored as an ordered chain of logs. The UI then uses metadata on each log to create grouped, readable trace views.

Topo Tracer does not start with a complex tree or graph model. It starts with a sequence and builds structure from that sequence.

---

## Core Concepts

Topo Tracer uses three main concepts:

```txt
Log
Group ID
Importance Level
```

### Log

A log is the main unit of the trace.

A log represents something that happened:

```txt
request received
user loaded
bar started
external service called
payment failed
response returned
```

Logs are connected in sequence using `sequenceIndex`, `previousLogId`, or `nextLogId`.

### Group ID

`groupId` tells Topo Tracer which visual group a log belongs to.

Example:

```txt
log1 groupId=foo
log2 groupId=foo
log3 groupId=bar
log4 groupId=external_service
```

The UI can render this as visual boxes:

```txt
[ foo ]
  log1
  log2

[ bar ]
  log3

[ external_service ]
  log4
```

### Importance Level

`importanceLevel` is a developer-facing field.

It controls how important or detailed a log is.

Suggested values:

```txt
0 = Core
1 = Important
2 = Normal
3 = Verbose
4 = Debug / Noisy
```

In the UI, this can be called **Visibility Level**.

The viewer controls how much detail they want to see using a single setting:

```txt
viewVisibility
```

Visibility rule:

```txt
show log if log.importanceLevel <= viewVisibility
hide log if log.importanceLevel > viewVisibility
```

Example:

```txt
viewVisibility = 1
```

Shows logs with:

```txt
importanceLevel 0
importanceLevel 1
```

Hides logs with:

```txt
importanceLevel 2+
```

---

## Basic Log Model

```ts
type TopoLog = {
  id: string
  traceId: string

  sequenceIndex: number
  previousLogId?: string
  nextLogId?: string

  groupId?: string
  groupName?: string

  severity: "debug" | "info" | "warn" | "error"
  message: string

  /**
   * Developer-facing visibility/detail control.
   *
   * 0 = Core
   * 1 = Important
   * 2 = Normal
   * 3 = Verbose
   * 4 = Debug / Noisy
   */
  importanceLevel: number

  timestamp: number
  attributes?: Record<string, unknown>
}
```

## How Grouping Works

Topo Tracer reads logs in order.

When consecutive logs have the same `groupId`, they belong to the same visual group.

Example:

```txt
log1 groupId=foo importanceLevel=0
log2 groupId=foo importanceLevel=2
log3 groupId=foo importanceLevel=3
```

Rendered at full visibility:

```txt
[ foo ]
  log1
  log2
  log3
```

Rendered at low visibility:

```txt
[ foo ]
  log1
```

Important rule:

> If logs have the same `groupId`, they stay in the same group. Different `importanceLevel` values inside the same group do not create nesting.

Inside the same group, `importanceLevel` only controls visibility.

---

## How Nesting Works

Nesting only happens when the `groupId` changes.

When Topo Tracer sees a new `groupId`, it compares the new log's `importanceLevel` with the current active group's structural importance.

### Higher Importance Number = Nested

```txt
log1 groupId=foo importanceLevel=0
log2 groupId=bar importanceLevel=1
```

Rendered:

```txt
[ foo ]
  [ bar ]
    log2
```

`bar` is nested because its importance number is higher.

Meaning:

```txt
foo = more important / higher-level
bar = more detailed / nested
```

### Same Importance Number = Next

```txt
log1 groupId=foo importanceLevel=0
log2 groupId=external_service importanceLevel=0
```

Rendered:

```txt
[ foo ] ---> [ external_service ]
```

Both groups are at the same visibility level, so they appear next to each other.

### Lower Importance Number = Move Back Up

```txt
log1 groupId=foo importanceLevel=0
log2 groupId=bar importanceLevel=1
log3 groupId=db importanceLevel=2
log4 groupId=external_service importanceLevel=0
```

Rendered:

```txt
[ foo ] ------------------> [ external_service ]
  [ bar ]
    [ db ]
```

`external_service` moves back up to level `0` and continues the main sequence.

---

## Group Structural Importance

A group's structural importance is taken from the first log that opens that group occurrence.

Example:

```txt
log1 groupId=foo importanceLevel=0
log2 groupId=foo importanceLevel=3
log3 groupId=foo importanceLevel=4
```

The group `foo` has structural importance `0`.

The later logs are still inside `foo`. Their `importanceLevel` only controls whether those logs are visible.

This gives one field two simple uses:

```txt
When groupId changes:
  importanceLevel helps decide group nesting/next behavior.

When groupId stays the same:
  importanceLevel controls log visibility inside the group.
```

---

## Resuming a Parent Group

A parent group can continue after a nested group.

Example:

```txt
log1 groupId=foo importanceLevel=0
log2 groupId=bar importanceLevel=1
log3 groupId=foo importanceLevel=3
```

Rendered:

```txt
[ foo ]
  log1
  [ bar ]
    log2
  log3
```

Rule:

> If a log's `groupId` matches an active ancestor group, Topo Tracer resumes that group instead of creating a new group occurrence.

---

## Reusing a Group ID Later

If a `groupId` appears again after the previous group occurrence has already closed, Topo Tracer treats it as a new visual occurrence.

Example:

```txt
log1 groupId=foo importanceLevel=0
log2 groupId=bar importanceLevel=0
log3 groupId=foo importanceLevel=0
```

Rendered:

```txt
[ foo ]
  log1

[ bar ]
  log2

[ foo ]
  log3
```

Internally, the UI can treat this as:

```txt
foo#1
bar#1
foo#2
```

Same `groupId`, different occurrence.

---

## Arrows

Arrows follow the log chain.

When a new group starts, the arrow comes from the previous active group.

Example:

```txt
log1 groupId=foo importanceLevel=0
log2 groupId=bar importanceLevel=1
log3 groupId=external_service importanceLevel=0
```

Rendered:

```txt
[ foo ]                         [ external_service ]
  |                                       ^
  +-- [ bar ] ----------------------------|
```

Meaning:

```txt
bar is nested inside foo
external_service is visually same-level as foo
bar triggered external_service
```

Rule:

```txt
Group placement comes from importanceLevel.
Arrow source comes from the previous active group in the log chain.
```

## Rendering Algorithm

Topo Tracer can render the UI using a group stack.

For every log in sequence:

```txt
1. If the log has the same groupId as the current active group:
     append it to the current group.

2. Else if the log.groupId matches an active ancestor group:
     close inner groups until that ancestor is active again.
     append the log to that ancestor group.

3. Else this log starts a new group occurrence.

4. Compare the new group's importanceLevel with the current active group's structural importance:

   - higher number:
       nest the new group inside the current group

   - same number:
       place the new group next to the current group

   - lower number:
       move back up until the correct level is reached,
       then place the new group there

5. Draw the arrow from the previous active group to the new group.

6. Apply visibility filtering:
     show log if log.importanceLevel <= viewVisibility
```

## Pagination

Pagination is simple because the source of truth is the ordered log chain.

Topo Tracer paginates logs, not visual groups.

Request shape:

```ts
getTraceLogsPage({
  traceId: string
  cursor?: string
  limit: number
  viewVisibility: number
})
```

Response shape:

```ts
type TraceLogsPage = {
  logs: TopoLog[]

  pageState: {
    openGroupsAtStart: OpenGroup[]
    openGroupsAtEnd: OpenGroup[]
  }

  hiddenCounts: {
    hiddenByVisibility: number
  }

  previousCursor?: string
  nextCursor?: string
}

type OpenGroup = {
  groupId: string
  groupName?: string
  structuralImportanceLevel: number
}
```

A page can begin or end in the middle of a group.

Example:

```txt
Page 1:
[ foo ]
  log1
  log2
  continues ->

Page 2:
<- continued from foo
[ foo ]
  log3
  log4
```

The backend returns `openGroupsAtStart` and `openGroupsAtEnd` so the UI can render continuation markers correctly.

Pagination rule:

```txt
1. Sort logs by sequenceIndex.
2. Apply viewVisibility.
3. Paginate the visible log chain.
4. Return openGroupsAtStart and openGroupsAtEnd.
5. UI reconstructs visual groups from groupId and importanceLevel.
```

---

## Errors and Warnings

Errors and warnings should still affect the group summary, even if the full log is hidden by visibility level.

Example:

```txt
log1 groupId=foo severity=error importanceLevel=4
```

At low visibility, the UI can still show:

```txt
[ foo ] 1 error
```

The full error log appears only when the viewer increases visibility.

---

## Final Rules

Topo Tracer follows these rules:

```txt
Logs are connected in sequence.
Each log can have a groupId.
Each log has an importanceLevel.
```

Grouping:

```txt
Same groupId:
  stay in the same group.
  importanceLevel only controls visibility.

Different groupId:
  start or resume a group.
  compare importanceLevel to decide nesting or next placement.
```

Placement:

```txt
Higher importanceLevel number:
  nested group.

Same importanceLevel:
  next group.

Lower importanceLevel number:
  move back up, then continue.
```

Visibility:

```txt
show log if log.importanceLevel <= viewVisibility
```

Pagination:

```txt
paginate the ordered log chain
reconstruct groups in the UI
use openGroupsAtStart/openGroupsAtEnd for continuation
```

---

## Summary

Topo Tracer is a log-chain-first tracing logger.

It keeps the data model simple:

```txt
log -> log -> log
```

Then it uses:

```txt
groupId
importanceLevel
```

to create grouped, readable trace views.

For developers:

```txt
groupId = where the log belongs
importanceLevel = how important or detailed the log is
```

For users:

```txt
Visibility Level = how much detail to show
```

This gives Topo Tracer sequencing, grouping, nesting, visibility control, arrows, and pagination without requiring a complex graph or span-tree model.
