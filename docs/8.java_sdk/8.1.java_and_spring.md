# 8. Java & Spring SDK

The Java SDK provides a high-performance, thread-safe client for Topo-Tracer, with specialized support for Spring Boot applications via auto-configuration and AOP.

## 1. Java SDK (Core)

The core Java SDK is built on `ThreadLocal` storage for context propagation and uses a background scheduler for non-blocking telemetry ingestion.

### 1.1. Context Propagation
The SDK uses `TraceContext` to maintain the active span across a single thread. For multi-threaded applications, the SDK provides wrappers for `ExecutorService` to ensure the trace context follows the execution flow.

### 1.2. Usage Example

```java
Tracer tracer = Tracer.builder()
    .endpoint("http://localhost:8787")
    .apiKey("your-api-key")
    .serviceName("inventory-service")
    .build();

// Programmatic tracing
tracer.trace("ProcessOrder", () -> {
    // Business logic here
    return "Order Confirmed";
});
```

## 2. Spring Boot Integration

The Spring SDK (`topo-tracer-spring-sdk`) leverages Spring Boot's auto-configuration to provide "zero-code" tracing for standard web applications.

### 2.1. Auto-Configuration
By adding the SDK to your classpath, Topo-Tracer automatically registers:
- **`TracingFilter`**: Captures incoming HTTP requests and extracts `X-Trace-Id` headers.
- **`TracingClientHttpRequestInterceptor`**: Automatically injects trace headers into outbound `RestTemplate` or `RestClient` calls.
- **`TracingTaskDecorator`**: Ensures trace context is propagated to `@Async` methods.

### 2.2. Annotation-Based Tracing (`@Traced`)
You can use the `@Traced` annotation on any Spring Bean method to create a new span automatically.

```java
@Service
public class OrderService {

    @Traced(name = "CheckInventory", type = "db")
    public void verifyStock() {
        // Automatically traced as a 'db' node
    }
}
```

### 2.3. Configuration Properties
Configure the SDK via `application.yml`:

```yaml
topotracer:
  endpoint: http://topo-tracer-server:8787
  api-key: ${TOPOTRACER_API_KEY}
  service-name: my-spring-app
  batch-size: 100
  flush-interval-ms: 5000
```

## 3. In-Span Logging

Topo-Tracer treats logs as first-class citizens in the trace graph. Instead of burying logs inside a span's metadata, `tracer.log()` creates a 0ms **Node** with `node_type: "log"`.

### Why Logs as Nodes?
- **Ordering:** Logs appear exactly where they happened in the flow, participating in the topological sort.
- **Filtering:** Logs can be filtered by `importance_level` (0 = Critical, 3 = Low).
- **Visualization:** Logs are rendered as distinct nodes, providing a clear timeline of events within a trace.

### Usage

```java
tracer.trace("process-payment", () -> {
    // Simple log
    tracer.log("Validating credit card");

    // Log with metadata
    tracer.log("Payment gateway responded", Map.of(
        "status", "success",
        "gateway", "stripe"
    ));

    // Log with importance level override (omitting metadata)
    tracer.log("Retrying connection", 1); // High importance

    // Log with both metadata and importance level
    tracer.log("Database write finished", Map.of("records", "15"), 2);
});
```

## 4. Technical Implementation Details

### Thread Safety
The SDK uses `ConcurrentHashMap` for attribute storage and `synchronized` blocks for buffer management, ensuring it can handle high-concurrency Spring environments without dropping telemetry.

### Performance
- **Non-blocking:** Telemetry is buffered in-memory and flushed by a daemon thread.
- **Batching:** Reduces network overhead by grouping multiple lifecycle events into a single HTTP POST.
- **Jittered Retries:** Prevents "thundering herd" issues when the Topo-Tracer server is under load.
