# 8. Java & Spring SDK

The Java SDK provides a high-performance, thread-safe client for Topo-Tracer, with specialized support for Spring Boot applications via auto-configuration, customizers, and AOP.

---

## 1. Java SDK (Core)

The core Java SDK is built on `ThreadLocal` storage for context propagation and uses a background executor pool for non-blocking telemetry ingestion.

### 1.1. Context Propagation
The SDK uses `TraceContext` to maintain the active span across a single thread. For multi-threaded applications, the SDK provides wrappers for `ExecutorService` to ensure the trace context follows the execution flow.

### 1.2. Non-Blocking Ingestion & Error Tolerance
- **Asynchronous Ingestion**: Ingest payloads (trace starts, node starts/ends, edge starts/ends) are buffered in memory and flushed asynchronously by a background `ExecutorService` rather than blocking the application thread.
- **Fail-Safety (`ignoreFailures`)**: When the Topo-Tracer server is offline or unreachable, background retry loops execute using exponential backoff. Ingestion errors are captured internally and logged without throwing exceptions or blocking application execution.

---

## 2. Customizable Hooks

The SDK provides synchronous logging and tracing hooks to let applications easily attach external tooling (e.g. SLF4J, metric collectors, or auditing systems) directly to tracing lifecycle events.

### 2.1. LogHook
Fired synchronously when a log node is created via `tracer.log(...)`.
```java
@FunctionalInterface
public interface LogHook {
    void onLog(String message, Map<String, String> data, Integer importanceLevel);
}
```

### 2.2. TraceHook
Fired synchronously when a span starts or finishes.
```java
public interface TraceHook {
    void onSpanStart(Span span);
    void onSpanEnd(Span span);
}
```

### 2.3. Spring Boot Registration
Simply declare beans of type `LogHook` or `TraceHook` in your Spring configuration class; the Spring Boot starter auto-configuration automatically discovers and registers them:
```java
@Configuration
@Slf4j
public class TracingHookConfiguration {

    @Bean
    public LogHook slf4jLogHook() {
        return (message, data, importanceLevel) -> {
            log.info("[Trace Log] {} | data: {} | importance: {}", message, data, importanceLevel);
        };
    }
}
```

## 3. Typed Class-Based Importance Levels

To support custom, autocomplete-friendly, and type-safe importance levels, the SDK uses an extensible class hierarchy instead of locked enums.

### 3.1. Extensible Class Hierarchy
- **`Importance`**: A base class holding an integer level and an optional string label.
- **`TypedImportance<E extends Enum<E>>`**: An abstract generic subclass extending `Importance` that links a custom enum type value to a level and label.
- **`DefaultImportance`**: Implements standard importance constants wrapping the deprecated `TopoImportance` enum (`CRITICAL`, `HIGH`, `MEDIUM`, `LOW`, `DYNAMIC`).

### 3.2. Defining a Custom Importance System
Developers can define their own scale using a custom enum and class:
```java
public enum CustomLevel {
    FATAL, WARNING, INFO
}

public class CustomImportance extends TypedImportance<CustomLevel> {
    public static final CustomImportance FATAL = new CustomImportance(CustomLevel.FATAL, 0, "Fatal Issue");
    public static final CustomImportance WARNING = new CustomImportance(CustomLevel.WARNING, 1, "Warning Alert");
    public static final CustomImportance INFO = new CustomImportance(CustomLevel.INFO, 2, "Info Details");

    private CustomImportance(CustomLevel value, int level, String label) {
        super(value, level, label);
    }
}
```

They can now pass `CustomImportance.FATAL` to `tracer.log()` or `TraceOptions.builder().importance()` with IDE autocompletion and compile-time checks.

## 4. Programmatic Customization (`TracerBuilderCustomizer`)

Instead of defining mappings across multiple places (properties and code), applications can register a `TracerBuilderCustomizer` bean to configure the tracer cleanly in code.

### 4.1. Usage Example
Configure custom importance levels and automatic node-type importance mapping:
```java
@Configuration
public class TracingHookConfiguration {

    @Bean
    public TracerBuilderCustomizer topoTracerCustomizer() {
        return builder -> {
            // Register custom class-based importance levels to populate trace header labels
            builder.importance(
                CustomImportance.FATAL,
                CustomImportance.WARNING,
                CustomImportance.INFO
            );

            // Automatically map standard node types to default levels in annotations
            builder.nodeTypeImportance("controller", 0);
            builder.nodeTypeImportance("service", 0);
            builder.nodeTypeImportance("db-call", 1);
            builder.nodeTypeImportance("io", 2);
            builder.nodeTypeImportance("method", 3);
        };
    }
}
```

### 4.2. Annotation-Based Tracing (`@Traced`)
Using standard AOP, methods annotated with `@Traced` will automatically respect the mappings defined in your customizer:
```java
@Service
public class OrderService {

    @Traced(value = "CheckInventory", nodeType = "service")
    public void verifyStock() {
        // Automatically traced as a service call with importance level 0
    }
}
```

## 5. Configuration Properties

Configure the SDK base properties via `application.properties` or `application.yml`:

```properties
topotracer.endpoint=http://localhost:8787
topotracer.apiKey=your-api-key
topotracer.serviceName=my-spring-app
topotracer.batchSize=100
topotracer.flushIntervalMs=5000
topotracer.ignoreFailures=true
```
