# Backend Schema And Queries

This doc explains why backend tables, queries, and projection code are shaped the
way they are. Source code lives in:

- `carno.js/src/infra/ClickHouseService.ts`
- `carno.js/src/services/log/RawEventRepository.ts`
- `carno.js/src/services/log/TraceReadModelBuilder.ts`
- `carno.js/src/services/log/ReadModelRepository.ts`
- `carno.js/src/services/log/LogService.ts`
- `carno.js/src/services/log/worker/TraceReadModelWorker.ts`

## Backend Shape

Backend has four layers:

1. Ingestion validates event shape and writes immutable raw events.
2. Raw repository appends events and finds traces needing materialization.
3. Materializer rebuilds read rows from raw events.
4. Read service projects materialized rows into graph windows for frontend.

The write path is append-only. Read tables can be rebuilt. This keeps SDK writes
simple and makes late events safe.

## Raw Event Table

Table: `topo_tracer.node_trace_events`

```sql
CREATE TABLE IF NOT EXISTS topo_tracer.node_trace_events (
  trace_id String,
  event_id String,
  entity_id String,
  entity_type LowCardinality(String),
  event_type LowCardinality(String),
  occurred_at_ms Int64,
  received_at_ms Int64,
  name Nullable(String),
  importance_level Nullable(Int32),
  parent_id Nullable(String),
  from_node_id Nullable(String),
  to_node_id Nullable(String),
  label Nullable(String),
  status Nullable(String),
  data String
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(toDateTime(received_at_ms / 1000))
ORDER BY (trace_id, received_at_ms, event_id);
```

Design:

- `MergeTree`, not `ReplacingMergeTree`, because raw events are source of truth
  and must not collapse old facts.
- `trace_id` first in `ORDER BY` because every materializer query is scoped to
  one trace.
- `received_at_ms` before `event_id` gives deterministic replay when SDK clocks
  skew or events arrive with equal source time.
- `event_id` is generated by SDK or backend and breaks ties.
- `occurred_at_ms` is source time and is used for durations.
- `received_at_ms` is backend ingest time and is used for replay/tie-breaks.
- `importance_level` is nullable because only node start events need it.
- `data` is JSON string because current ClickHouse client path stores flexible
  metadata without schema migrations.

Write query:

```ts
client.insert({
  table: "topo_tracer.node_trace_events",
  values: events.map(...),
  format: "JSONEachRow",
});
```

No update/delete path exists for raw events.

## Materialization Picker Query

The worker asks raw repository which traces have new raw events:

```sql
SELECT raw.trace_id
FROM (
  SELECT trace_id, max(received_at_ms) AS latest_event_at
  FROM topo_tracer.node_trace_events
  GROUP BY trace_id
) AS raw
LEFT JOIN (
  SELECT trace_id, max(materialized_at_ms) AS latest_materialized_at
  FROM topo_tracer.node_trace_summary
  GROUP BY trace_id
) AS summary USING trace_id
WHERE latest_materialized_at IS NULL OR latest_event_at > latest_materialized_at
ORDER BY latest_event_at ASC
LIMIT {limit}
```

Why:

- `max(received_at_ms)` detects late raw writes.
- `max(materialized_at_ms)` detects latest completed projection.
- `LEFT JOIN` includes never-materialized traces.
- `ORDER BY latest_event_at ASC` drains older traces first.
- `LIMIT` protects worker loop from spending one tick on too many traces.

Current tradeoff:

- This query scans grouped raw/summary tables. Fine for dev and early scale.
- Later, add a small dirty-trace queue table if raw volume grows enough.

## Raw Replay Query

For one trace:

```sql
SELECT *
FROM topo_tracer.node_trace_events
WHERE trace_id = {traceId:String}
ORDER BY received_at_ms ASC, event_id ASC
```

Why:

- `WHERE trace_id` matches first `ORDER BY` column.
- `received_at_ms, event_id` gives stable replay.
- Causal ordering is computed after replay. Timestamps do not override graph
  parent/edge causality.

## Read Node Table

Table: `topo_tracer.node_read_nodes`

```sql
CREATE TABLE IF NOT EXISTS topo_tracer.node_read_nodes (
  trace_id String,
  id String,
  parent_id Nullable(String),
  name String,
  importance_level Int32,
  status LowCardinality(String),
  started_at_ms Nullable(Int64),
  ended_at_ms Nullable(Int64),
  duration_ms Nullable(Int64),
  ancestry_path Array(String),
  indent_level Int32,
  flow_order Int64,
  diagnostics Array(String),
  data String,
  materialized_at_ms Int64
) ENGINE = ReplacingMergeTree(materialized_at_ms)
PARTITION BY sipHash64(trace_id) % 32
ORDER BY (trace_id, flow_order, id);
```

Design:

- `ReplacingMergeTree(materialized_at_ms)` lets a later materializer run replace
  older read rows in `FINAL` queries.
- `PARTITION BY sipHash64(trace_id) % 32` spreads traces without depending on
  time. Reads are trace-scoped, so trace hashing keeps one trace bounded.
- `ORDER BY (trace_id, flow_order, id)` serves graph reads in causal order.
- `importance_level` is a first-class column because slider filtering is core.
- `indent_level` is stored so frontend layout does not recompute ancestry.
- `ancestry_path` is stored because ghost-node projection needs nearest visible
  ancestor lookup.
- `diagnostics` is an array so UI can show safety flags without parsing JSON.
- `data` remains JSON string for flexible user metadata.

Read query:

```sql
SELECT *
FROM topo_tracer.node_read_nodes FINAL
WHERE trace_id = {traceId:String}
ORDER BY flow_order ASC
```

Why `FINAL`:

- `ReplacingMergeTree` deduplicates during merges eventually.
- `FINAL` forces newest row for correctness in dev/read API.

Tradeoff:

- `FINAL` is more expensive. For larger scale, use versioned primary keys or
  materialized snapshots that avoid `FINAL` on hot graph reads.

## Read Edge Table

Table: `topo_tracer.node_read_edges`

```sql
CREATE TABLE IF NOT EXISTS topo_tracer.node_read_edges (
  trace_id String,
  id String,
  from_node_id String,
  to_node_id String,
  label String,
  status LowCardinality(String),
  started_at_ms Nullable(Int64),
  ended_at_ms Nullable(Int64),
  duration_ms Nullable(Int64),
  diagnostics Array(String),
  data String,
  materialized_at_ms Int64
) ENGINE = ReplacingMergeTree(materialized_at_ms)
PARTITION BY sipHash64(trace_id) % 32
ORDER BY (trace_id, from_node_id, to_node_id, id);
```

Design:

- Edges are loaded by trace and then lifted to visible/ghost endpoints in app
  code.
- Endpoint columns are ordered after `trace_id` because future queries may fetch
  edges by visible node window.
- `ended_at_ms` is nullable. Open edge means async/fire-and-forget and is valid.

Read query:

```sql
SELECT *
FROM topo_tracer.node_read_edges FINAL
WHERE trace_id = {traceId:String}
```

## Ancestry Index Table

Table: `topo_tracer.node_read_node_ancestry`

```sql
CREATE TABLE IF NOT EXISTS topo_tracer.node_read_node_ancestry (
  trace_id String,
  node_id String,
  ancestor_id String,
  ancestor_depth Int32,
  materialized_at_ms Int64
) ENGINE = ReplacingMergeTree(materialized_at_ms)
PARTITION BY sipHash64(trace_id) % 32
ORDER BY (trace_id, ancestor_id, node_id);
```

Design:

- One row per ancestor hop.
- `ancestor_depth` means index inside `ancestry_path`; it is not importance.
- `ORDER BY (trace_id, ancestor_id, node_id)` supports future descendant summary
  queries for ghost nodes:

```sql
SELECT node_id
FROM topo_tracer.node_read_node_ancestry FINAL
WHERE trace_id = {traceId:String}
  AND ancestor_id = {ancestorId:String}
```

Current code does not yet use this table for graph projection. It writes it now
so future large-trace ghost summaries can avoid loading every node.

## Trace Summary Table

Table: `topo_tracer.node_trace_summary`

```sql
CREATE TABLE IF NOT EXISTS topo_tracer.node_trace_summary (
  trace_id String,
  created_at_ms Int64,
  updated_at_ms Int64,
  node_count UInt64,
  edge_count UInt64,
  error_count UInt64,
  diagnostic_count UInt64,
  max_importance_level Int32,
  materialized_at_ms Int64
) ENGINE = ReplacingMergeTree(materialized_at_ms)
ORDER BY trace_id;
```

Design:

- Trace list and slider metadata should not load graph rows.
- `max_importance_level` sizes frontend slider.
- `node_count`, `edge_count`, `error_count`, and `diagnostic_count` give cheap
  overview.

Trace list query:

```sql
SELECT *
FROM topo_tracer.node_trace_summary FINAL
ORDER BY updated_at_ms DESC
LIMIT {limit} OFFSET {offset}
```

Count query:

```sql
SELECT count() AS total
FROM topo_tracer.node_trace_summary FINAL
```

Summary query:

```sql
SELECT *
FROM topo_tracer.node_trace_summary FINAL
WHERE trace_id = {traceId:String}
ORDER BY materialized_at_ms DESC
LIMIT 1
```

## Materializer Algorithm

Input: raw events ordered by backend receive time.

Steps:

1. Fold node events into `NodeDraft` by `entityId`.
2. Fold edge events into `EdgeDraft` by `entityId`.
3. Merge metadata from repeated events into `data`.
4. Use earliest started event as start time.
5. Use latest ended event as end time.
6. Compute duration from source occurred time.
7. Compute ancestry from `parentId`.
8. Compute causal `flowOrder` from parent links and edges.
9. Emit read nodes, read edges, ancestry rows, and summary row.

Node lifecycle:

- Missing node start -> `missingStart`
- Missing node end -> `missingEnd`
- End before start -> `negativeDuration`

Edge lifecycle:

- Missing edge start -> `missingStart`
- Missing edge end -> valid open async edge, not diagnostic.
- End before start -> `negativeDuration`
- Missing endpoint -> `orphanEdge`

Ancestry:

- Root node has `ancestryPath = []` and `indentLevel = 0`.
- Child path is parent path plus parent id.
- Cycle while walking parents -> `cycleDetected`.
- Parent id missing from trace -> `orphanNode`.

Flow order:

- Build graph adjacency from parent links and explicit edges.
- Topologically sort nodes.
- Tie-break by `startedAtUnixMs`, then id.
- If cycles remain, mark `cycleDetected` and append cyclic nodes after sorted
  nodes.

## Graph Projection Query Path

`GET /telemetry/traces/:traceId/graph` does:

1. Load summary.
2. Load all materialized nodes for trace.
3. Load all materialized edges for trace.
4. Apply importance projection in `LogService`.
5. Apply cursor window.

Current projection is in application code because ghost-node construction needs
graph semantics, not simple SQL filtering.

Projection rules:

- Visible node: `importanceLevel <= maxImportance`.
- Hidden node: `importanceLevel > maxImportance`.
- Hidden nodes group under nearest visible ancestor.
- Group with no visible ancestor becomes root ghost.
- Ghost node id is `ghost:{ancestorId}`.
- Ghost `importanceLevel` is `maxImportance + 1`.
- Ghost `indentLevel` is `parent.indentLevel + 1`.
- Ghost `flowOrder` is first hidden child order plus `0.1`.
- Ghost data stores first 25 hidden node ids and truncated count.

Frontend layout rules:

- Read API returns causal `flowOrder` and `indentLevel`.
- Frontend uses capped `indentLevel` as X axis.
- Frontend uses `flowOrder` order as Y axis.
- Downward or rightward edges may render as SVG arrows.
- Return-to-parent, backward, or cross-window edges may render as local chips by
  frontend instead of SVG arrows. Backend does not need separate edge shape data
  for this.
- Parent/child `continues` edges generated by SDK may be suppressed by frontend
  because parent links already place nodes into structural columns and can be
  rendered as quiet scope lines.

Edge lifting:

- Visible endpoint stays as-is.
- Hidden endpoint resolves to nearest ancestor ghost.
- If endpoint resolves to itself or leaves current window, edge is omitted.
- Multiple lifted edges with same `from->to:label` are collapsed.
- Collapsed edge records `hiddenEdgeCount`.

## Pagination

Request params:

- `maxImportance`: max visible importance threshold.
- `limit`: max projected nodes to return, default `250`, hard max `500`.
- `cursor`: base64url-encoded projected offset.

Response metadata:

- `hasBefore`
- `hasAfter`
- `previousCursor`
- `nextCursor`
- `returnedNodeCount`
- `totalNodeCount`
- `hiddenNodeCount`
- `ghostNodeCount`

Why offset cursor is acceptable now:

- Cursor applies after deterministic projection sort.
- It is simple and stable enough for current single-trace dev workflows.
- For very large mutable traces, future cursor should include materialized
  version and anchor flow order.

## Safety Nets

Implemented:

- Append-only raw writes.
- Read cap of `500` nodes.
- Importance threshold clamps between `0` and trace max.
- Invalid event shape rejected before insert.
- Non-negative `importanceLevel` enforced.
- Open async edges allowed.
- Diagnostics stored on read rows.
- Large traces never return full graph by default.

Known tradeoffs:

- Current graph endpoint loads all read nodes/edges for a trace before
  projection. This is acceptable for current dev traces and verified 10k nodes,
  but not final for very large production traces.
- `node_read_node_ancestry` is already written so next optimization can query
  hidden subtree summaries directly.
- `FINAL` queries keep correctness simple now. Later, use snapshot/versioned
  tables to reduce read cost.

## Change Checklist

When changing backend schema or projection:

1. Update `ClickHouseService.ts`.
2. Update repository insert/read mappings.
3. Update `types.ts`.
4. Update `TRACE_DESIGN.md`.
5. Update this doc.
6. Run backend build.
7. Run seed and SDK examples.
8. Verify graph window includes `hasBefore`, `hasAfter`, ghosts, and timing.
