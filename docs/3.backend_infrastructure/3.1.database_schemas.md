# 3.1. Database Schemas

Topo-Tracer uses a hybrid database approach to balance high-volume telemetry ingestion with reliable relational data management.

## 1. PostgreSQL: System & Relational Data

PostgreSQL is the source of truth for user accounts, authentication, and the transactional outbox.

### 1.1. Core Tables

- **`users`**: Active user profiles.
    - `auth_version`: Security versioning for JWT invalidation.
- **`pending_users`**: Temporary storage for signup flows (pre-OTP verification).
- **`token_otps`**: Short-lived verification codes for signup and password resets.
- **`api_keys`**: Cryptographically hashed keys for SDK authentication.
- **`outbox_events`**: Atomic storage for Event Bus signals to ensure delivery consistency.

### 1.2. Transactional Integrity
We use Postgres transactions for all `AuthService` operations. If a signup fails, the `outbox_event` is never written, preventing "phantom" emails or notifications.

## 2. ClickHouse: Telemetry & Analytical Data

ClickHouse handles the heavy lifting of storing millions of traces and powering the visualization engine.

### 2.1. Raw Telemetry (`MergeTree`)
Append-only tables for high-speed writes from SDKs.
- `node_events`: Start/End lifecycle events.
- `edge_events`: Parent/Child connection events.
- `trace_events`: Meta updates (naming, labeling).

### 2.2. Materialized Read Model (`ReplacingMergeTree`)
Optimized for the Trace Explorer UI. These tables store the **latest collapsed state** of every node and edge.
- `read_nodes`: Includes pre-calculated `flow_order` and `duration`.
- `read_edges`: Includes denormalized flow orders for both source and target nodes.
- `trace_summaries`: Aggregated statistics (node counts, time bounds) and health diagnostics.

### 2.3. Progress Tracking
- `materialization_checkpoints`: Stores exact byte/timestamp offsets to ensure materialization resumes correctly after a crash.

---

## 3. Deployment & Dev Workflow
Schema definitions are centralized in `hono-server/src/infra/db/`. In development, the server automatically bootstraps tables using `CREATE TABLE IF NOT EXISTS` statements.

### 3.1. Schema flattening
To keep the codebase lean during rapid development, we do not use separate migration files. Instead:
1.  Update the `CREATE TABLE` string in `infra/db/<postgres|clickhouse>/schema.ts`.
2.  The server verifies and updates the schema on startup.
