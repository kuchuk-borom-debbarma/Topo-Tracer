# 3.1. Database Schemas

Topo-Tracer uses Postgres for relational application state and ClickHouse for telemetry storage and read models.

Schema definitions live in:

- `hono-server/src/infra/db/postgres/schema.ts`
- `hono-server/src/infra/db/clickhouse/schema.ts`

The backend runs `CREATE TABLE IF NOT EXISTS` bootstrap statements on startup.

## Postgres

Postgres stores auth and outbox data.

| Table | Purpose |
| --- | --- |
| `users` | Active users, unique emails, password hashes, and `auth_version` for JWT invalidation. |
| `pending_users` | Signup records waiting for OTP verification, including optional trace context. |
| `token_otps` | OTP rows for signup and password reset flows. |
| `api_keys` | SDK API keys by user, including hash, prefix, optional stored key value, timestamps, and revocation. |
| `outbox_events` | Pending, processing, and sent event-bus publications for transactional workflows. |

`PgOutboxStore` uses `FOR UPDATE SKIP LOCKED` when claiming pending rows so multiple relay instances can work without claiming the same event.

## ClickHouse Raw Event Tables

Raw events are append-only facts.

### `trace_events`

Trace-level metadata:

- `user_id`;
- `trace_id`;
- `event_type`;
- optional `name`;
- `importance_labels`;
- `timestamp_ms`.

Engine:

```sql
ENGINE = MergeTree
ORDER BY (user_id, trace_id, timestamp_ms)
```

### `node_events`

Node lifecycle events:

- `id`;
- `user_id`;
- `trace_id`;
- `event_type`: `0` start, `1` end;
- nullable `started_at_ms`;
- nullable `ended_at_ms`;
- nullable `node_type`;
- `data Map(String, String)`;
- nullable `message`;
- nullable `importance_level`;
- nullable `name`;
- nullable `group_parent_id`;
- nullable `layer_key`, `layer_label`, and `layer_order`.

Engine:

```sql
ENGINE = MergeTree
ORDER BY (user_id, trace_id, id, event_type)
```

### `edge_events`

Edge lifecycle events:

- `id`;
- `user_id`;
- `trace_id`;
- `event_type`: `0` start, `1` end;
- nullable `started_at_ms`;
- nullable `ended_at_ms`;
- nullable `edge_type`;
- nullable `from_node_id`;
- nullable `to_node_id`;
- `data Map(String, String)`.

Engine:

```sql
ENGINE = MergeTree
ORDER BY (user_id, trace_id, id, event_type)
```

## ClickHouse Read Model Tables

Read tables store versioned latest state. They use `ReplacingMergeTree`, but query code still uses `argMax(..., materialized_at_ms)` to read the latest version deterministically.

### `read_nodes`

One logical row per latest node version:

- identity: `id`, `user_id`, `trace_id`;
- graph fields: `importance_level`, `flow_order`;
- time fields: `started_at_ms`, `ended_at_ms`, `original_started_at_ms`, `clock_skew_ms`;
- display fields: `node_type`, `name`, `start_message`, `end_message`;
- grouping fields: `group_parent_id`, `layer_key`, `layer_label`, `layer_order`;
- payload: `data`;
- version: `materialized_at_ms`.

Engine:

```sql
ENGINE = ReplacingMergeTree(materialized_at_ms)
ORDER BY (user_id, trace_id, id)
```

### `read_edges`

One logical row per latest edge version:

- identity: `id`, `user_id`, `trace_id`;
- graph fields: `from_node_id`, `to_node_id`, `from_flow_order`, `to_flow_order`, `edge_type`;
- time fields: `started_at_ms`, `ended_at_ms`, `original_started_at_ms`, `clock_skew_ms`;
- payload: `data`;
- version: `materialized_at_ms`.

Engine:

```sql
ENGINE = ReplacingMergeTree(materialized_at_ms)
ORDER BY (user_id, trace_id, id)
```

### `trace_summaries`

Trace-level materialized metadata:

- name and importance labels;
- node and edge counts;
- min and max importance;
- trace time bounds;
- materialization timestamp;
- diagnostics counters.

Engine:

```sql
ENGINE = ReplacingMergeTree(materialized_at_ms)
ORDER BY (user_id, trace_id)
```

### `materialization_checkpoints`

Exact replay progress per trace:

- `trace_progress_timestamp`;
- `node_progress_timestamp`, `node_progress_id`, `node_progress_event_type`;
- `edge_progress_timestamp`, `edge_progress_id`, `edge_progress_event_type`;
- `updated_at_ms`.

Engine:

```sql
ENGINE = ReplacingMergeTree(updated_at_ms)
ORDER BY (user_id, trace_id)
```

## Realtime Summary Tables

ClickHouse also defines:

- `trace_summaries_realtime`;
- `node_events_summary_mv`;
- `edge_events_summary_mv`.

The materialized views aggregate raw event counts and time bounds into an `AggregatingMergeTree`. `LogReadRepoClickHouse.loadTraceSummary` combines realtime counts/bounds with worker-computed diagnostics from `trace_summaries`.

## Schema Evolution

Grouping columns are added with idempotent `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` statements during ClickHouse bootstrap. This keeps existing local databases usable without introducing a separate migration runner.

## Deletion

`DELETE /api/v1/traces/:traceId` calls `LogReadRepoClickHouse.deleteTrace`, which submits asynchronous ClickHouse mutations against raw, read, summary, checkpoint, and realtime summary tables:

```sql
ALTER TABLE <table>
DELETE WHERE user_id = {userId:String} AND trace_id = {traceId:String}
SETTINGS mutations_sync = 0
```

The endpoint returns `202` after ClickHouse accepts the mutations.
