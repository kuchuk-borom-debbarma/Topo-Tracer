# Database Schemas

Topo-Tracer uses a polyglot persistence architecture. **Postgres** handles highly relational, strictly consistent domain entities (like users, API keys, and outbox queues). **ClickHouse** handles high-throughput, append-heavy telemetry data (like raw events and materialized graphs).

## 1. Postgres Schema (`src/infra/db/postgres/schema.ts`)

Postgres uses Kysely for type-safe query building. It owns the following core tables:

### Users & Auth
-   `users`: Core identity (`id`, `username`, `email`, `password_hash`).
-   `pending_users`: Ephemeral store for signups awaiting OTP verification.
-   `user_otps`: Time-limited one-time passwords for signup and password resets.
-   `sessions`: Active user login sessions.
-   `api_keys`: Long-lived tokens for SDK ingestion without requiring interactive login.

### Outbox
-   `outbox_events`: The transactional outbox queue (`id`, `topic`, `payload`, `status`, `created_at`). Events are written here synchronously during trace ingestion, and then relayed asynchronously to the Event Bus to ensure guaranteed delivery.

## 2. ClickHouse Schema (`src/infra/db/clickhouse/schema.ts`)

ClickHouse stores the telemetry. Because ClickHouse prefers wide, flat tables and append-only operations, we split the design into "Raw Events" and "Read Models".

### Raw Events (`node_trace_events`)
**Engine**: `MergeTree()`
**Purpose**: Stores every immutable lifecycle event emitted by the SDK (`node.started`, `edge.ended`). 
-   `trace_id`, `event_id`, `entity_id`
-   `entity_type`, `event_type`
-   `occurred_at_ms`, `received_at_ms`
-   `data` (JSON string)

### Read Models (`node_read_nodes` & `node_read_edges`)
**Engine**: `ReplacingMergeTree(materialized_at_ms)`
**Purpose**: Stores the computed, collapsed state of a node or edge.
-   When `TraceReadModelMaterializer` runs, it inserts rows here. 
-   If a node is updated (e.g., it goes from `open` to `ended`), a new row is inserted with a higher `materialized_at_ms`.
-   The `ReplacingMergeTree` engine will eventually merge out the older rows in the background. 
-   At query time, the API uses `argMax(..., materialized_at_ms)` grouped by `id` to ensure the absolute latest state is always returned, even before the background merge completes.

### Alias Rule
When querying read models, we must never alias aggregate outputs back to their source column names inside the same aggregate `SELECT`. We use names like `latest_materialized_at_ms` and map them to API field names in TypeScript to avoid ClickHouse alias substitution creating illegal nested aggregates.
