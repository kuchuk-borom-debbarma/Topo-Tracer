# 3.2. Event Bus and Idempotency

Topo-Tracer treats async delivery as at-least-once. The event bus API is intentionally small, but every event carries enough metadata for routing, ordering, observability, and deduplication.

## Contract

`IEventBus` lives at `hono-server/src/infra/event-bus/api/IEventBus.ts`.

```ts
publish(events, options?): Promise<void>
subscribe(options, handler): Promise<void>
```

Published events include:

- `topic`: stable event name;
- `idempotencyId`: stable identity for the same logical work;
- `key`: ordering lane, usually `userId` or `traceId`;
- `data`: small JSON-compatible payload.

Subscribe options include:

- `topic`;
- `consumerName`;
- `batchSize`.

Handlers receive arrays because real brokers commonly deliver batches.

## Implementations

### `DevEventBus`

`DevEventBus` is in-process and used unless `EVENT_BUS_TYPE=kafka`.

It:

- stores handlers by topic;
- groups published events by topic;
- delivers batches to registered handlers;
- filters duplicate `idempotencyId` values within a batch;
- asks the configured `IIdempotencyStore` whether a consumer already processed an event;
- marks events processed only after the handler succeeds.

It is useful for local development, but it is not durable across process restarts.

### `KafkaEventBus`

`KafkaEventBus` uses `kafkajs`.

It:

- creates an idempotent producer;
- sends messages keyed by event `key`;
- stores `idempotencyId` in message headers;
- subscribes by `consumerName` as the Kafka consumer group id;
- parses message JSON;
- applies the same consumer-side idempotency filtering before calling handlers;
- exposes `shutdown` for graceful producer/consumer disconnect.

Configure it with:

```txt
EVENT_BUS_TYPE=kafka
KAFKA_BROKERS=localhost:9092
```

## Idempotency Store

The default singleton is `CacheIdempotencyStore(cache)`, exported from `src/infra/event-bus/idempotency`.

Available implementations:

- `CacheIdempotencyStore`: uses the configured `ICache`.
- `InMemoryIdempotencyStore`: local `Map` with 24-hour expiration.

Idempotency is scoped by `consumerName`, so two different consumers can process the same published event independently.

## Outbox

The outbox lives under `src/infra/event-bus/outbox`.

The default store is `PgOutboxStore`, backed by the Postgres `outbox_events` table. `OutboxRelay`:

1. recovers stuck `processing` rows;
2. claims pending rows with `FOR UPDATE SKIP LOCKED`;
3. publishes them to the bus with `bypassOutbox: true`;
4. marks them `sent` on success;
5. returns them to `pending` on publish failure;
6. backs off with jitter after failures.

Auth signup currently uses transactional publishing through this outbox path. Telemetry ingestion currently publishes directly to the bus and lets `LogIngestConsumer` write raw ClickHouse events asynchronously.

## Topics In Use

| Topic | Producer | Consumer | Purpose |
| --- | --- | --- | --- |
| `log.telemetry.received` | `LogServiceImpl` and self-tracing middleware | `LogIngestConsumer` | Carry raw telemetry batches to the ClickHouse writer. |
| `log.trace.ingested` | `LogIngestConsumer` | `ReadOptimisedAggregator` | Tell the materializer a trace has new raw events. |
| `auth.signup.started` | `AuthServiceImpl` | `AuthEventConsumer` | Send signup OTP notification asynchronously. |

## Backend Rules

- Always publish stable `idempotencyId` values.
- Use `traceId` as the key for trace rebuild work.
- Keep event payloads small.
- Do not mark idempotency before handler success.
- Make consumers safe to rerun.
- Do not rely on delivery exactly once.
