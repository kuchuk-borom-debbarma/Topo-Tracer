# Event Bus & Idempotency

Topo-Tracer is a reactive system. When events are ingested or when users sign up, those workflows publish events to an `IEventBus` which triggers background consumers.

## 1. The Event Bus Contract (`IEventBus`)

The event bus handles batch-native delivery:
-   `publish(events)` accepts an array of events.
-   `subscribe(topic, handler)` delivers arrays to the handler.

The system relies on the bus for:
-   **Routing**: Topics like `trace.events.ingested` or `auth.signup.started`.
-   **Ordering**: `key` (like a `traceId`) ensures events for the same trace are delivered sequentially.
-   **Idempotency**: `idempotencyId` prevents the same event from being processed twice.

### Implementations
-   `KafkaEventBus`: Intended for production. Provides durable, partitioned, and highly available delivery.
-   `DevEventBus`: In-memory development implementation that simulates Kafka partitions, batching, and ordering using Node timers and arrays.

## 2. Idempotency Store (`IIdempotencyStore`)

Because distributed event buses (like Kafka) usually provide at-least-once delivery, consumers might receive the same batch of events twice (e.g., due to a network timeout during acknowledgment).

To prevent rebuilding the read models unnecessarily, the `LogIngestConsumer` and `AuthEventConsumer` wrap their logic in idempotency checks using the `IIdempotencyStore`.

1.  Before processing, the consumer checks if `idempotencyId` exists.
2.  If it does, the event is skipped.
3.  If it doesn't, the event is processed and the `idempotencyId` is stored.

### Implementations
-   `CacheIdempotencyStore`: Uses Redis (or another `ICache`) to store the ID with a TTL.
-   `InMemoryIdempotencyStore`: Used locally, stores IDs in an LRU Map.
