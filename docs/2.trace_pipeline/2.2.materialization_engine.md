# 2.2. Materialization Engine

Materialization converts raw lifecycle events into read-optimized graph state. This is where Topo-Tracer pays the cost of folding starts/ends, ordering the graph, detecting diagnostics, and creating compact rows for trace reads.

Main files:

- `hono-server/src/services/log/internal/worker/ReadOptimisedAggregator.ts`
- `hono-server/src/services/log/internal/materialization/TraceReadModelMaterializer.ts`
- `hono-server/src/services/log/internal/materialization/flowOrder.ts`
- `hono-server/src/services/log/internal/repo/impl/LogReadRepoClickHouse.ts`

## Trigger

`ReadOptimisedAggregator` subscribes to `log.trace.ingested`.

For each delivered batch it:

- ignores invalid payloads;
- deduplicates repeated work by `traceId` inside the batch;
- serializes rebuilds for the same `{userId, traceId}` across batches;
- uses `p-limit` to cap concurrent materializations at 10 per worker instance.

This avoids redundant rebuilds and prevents one hot trace from racing itself.

## Incremental Replay

`TraceReadModelMaterializer.materializeTrace` performs an incremental rebuild:

1. Load the latest `materialization_checkpoints` row for the trace.
2. Load the latest existing read model from `read_nodes`, `read_edges`, and `trace_summaries`.
3. Load raw `trace_events`, `node_events`, and `edge_events` after the checkpoint.
4. Fold node and edge start/end events into maps keyed by id.
5. Compute deterministic `flowOrder` from explicit edges.
6. Drop orphan edges from saved read edges.
7. Correct causal clock skew.
8. Build a `trace_summaries` row with counts and diagnostics.
9. Save read-model rows.
10. Save the next checkpoint.

The order matters: the read model is saved before the checkpoint. If the process crashes after saving read rows but before saving the checkpoint, the next run replays the same raw events and overwrites the same logical node/edge state.

## Checkpoints

Checkpoints store the raw event progress for one user-owned trace:

- latest trace metadata timestamp;
- latest node event timestamp, id, and event type;
- latest edge event timestamp, id, and event type;
- checkpoint write timestamp.

The materializer does not infer progress from `read_nodes` or `read_edges`.

`LogReadRepoClickHouse.loadRawEventsAfterCheckpoint` applies a 10-second lookback window to node and edge checkpoint timestamps. That intentionally replays a small overlap so same-millisecond or late-arriving events are not skipped by tuple comparison boundaries.

See [9.1. ClickHouse Checkpoint Lookback Journal](../9.development_journal/9.1.clickhouse_checkpoint_lookback.md) for the bug that motivated this behavior.

## Event Folding

Node start events create or update `ReadNode` entries with:

- `nodeType`;
- `data`;
- optional display `name`;
- `startedAt` and `originalStartedAt`;
- `importanceLevel`;
- `startMessage`.

Node end events update:

- `endedAt`;
- `endMessage`.

Edge start events create or update `ReadEdge` entries with:

- `edgeType`;
- `fromNodeId`;
- `toNodeId`;
- `startedAt` and `originalStartedAt`;
- `data`.

Edge end events update `endedAt`.

## Flow Order

`computeFlowOrder` uses a modified Kahn algorithm:

- builds adjacency and in-degree maps from explicit read edges;
- ignores orphan edges while counting them as diagnostics;
- uses a priority queue ordered by `startedAt` and `id` for deterministic root selection;
- detects self-edges and unresolved cycles;
- breaks cycles by forcing remaining unprocessed nodes into the queue.

The output is a `Map<nodeId, flowOrder>`. Edges store `fromFlowOrder` and `toFlowOrder` to support projection queries without recomputing graph order.

## Clock-Skew Correction

After flow ordering, the materializer corrects causal timestamp violations.

For every node in flow order:

- reset `startedAt` to `originalStartedAt`;
- find parent nodes from saved edges;
- if the child starts before the earliest parent start, shift the child to `minParentStart + 1`;
- shift `endedAt` by the same delta to preserve duration;
- record `clockSkewMs` and increment `diagClockSkew`.

Edges are then aligned with their corrected source node when necessary.

## Diagnostics

Trace summaries contain:

- `diagMissingStarts`: end events without matching starts.
- `diagMissingEnds`: nodes or edges that never ended.
- `diagNegativeDurations`: end timestamp earlier than start timestamp.
- `diagCycles`: cycles detected during flow ordering.
- `diagOrphanEdges`: edges that reference missing nodes.
- `diagInvalidImportance`: invalid or missing importance/start data.
- `diagClockSkew`: causal timestamp corrections.
- `diagLimitExceeded`: trace size or depth limits exceeded.

These diagnostics are visible through trace summaries and the frontend.

## Read Tables

The materializer writes:

- `read_nodes`;
- `read_edges`;
- `trace_summaries`;
- `materialization_checkpoints`.

The read tables use `ReplacingMergeTree(materialized_at_ms)`. Reads use `argMax(..., materialized_at_ms)` to select the latest version without relying on `FINAL`.
