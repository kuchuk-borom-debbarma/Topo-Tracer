# 2.2. Trace Materialization & ClickHouse Schema

This document explains the deep-level mechanics of how Topo-Tracer transforms raw, disordered telemetry into a high-performance, topologically-sorted read model using ClickHouse.

## 1. The Core Problem: Telemetry Chaos
Telemetry data from distributed systems is inherently messy:
- **Out of Order:** Network lag means a "Node End" event might arrive before its "Node Start".
- **Disordered Clocks:** Different servers have clock skew; a child span might appear to start *before* its parent.
- **High Volume:** Millions of events per second make traditional relational JOINs too slow for real-time visualization.

## 2. The Solution: Multi-Stage Materialization
Topo-Tracer uses a "Lambda-lite" architecture. We store raw events in append-only tables and asynchronously "materialize" them into a read-optimized state.

### Phase A: Append-Only Ingestion
When the SDK sends a batch, the `LogIngestConsumer` writes raw data to:
- `node_events`: Every start/end event for a span.
- `edge_events`: Every connection between spans.
- `trace_events`: Trace-level metadata (names, labels).

**Storage Engine:** `MergeTree`. This is ClickHouse's standard high-performance engine for heavy writes.

### Phase B: Reactive Signal
After writing raw data, a `log.trace.ingested` event is published to the Event Bus (Kafka). This signals that a specific `trace_id` has new data and needs a "rebuild".

### Phase C: The Materialization Engine
The `ReadOptimisedAggregator` receives the signal and invokes the `TraceReadModelMaterializer`. This is where the magic happens:

1.  **Deduplication:** Uses `ReplacingMergeTree` to keep only the latest "version" of a span if multiple updates occur.
2.  **Topological Sorting:** Computes the `flow_order`. This ensures that when the frontend renders, it knows exactly which node comes after which, even if timestamps are messy.
3.  **Clock Skew Correction:** If a child starts at `T=10` but its parent starts at `T=12`, the materializer "heals" the timeline by shifting the child to `T=13`.
4.  **Denormalization:** It pre-calculates everything the UI needs (duration, child counts, importance levels) and saves it to the **Read Tables**.

## 3. ClickHouse Schema Design

### 3.1. Raw Event Tables (`node_events`, `edge_events`)
These are optimized for **write throughput**.
- **Immutable:** Once a row is written, it never changes.
- **Granular:** One row per start, one row per end.
- **Ordering:** `ORDER BY (user_id, trace_id, id, event_type)`. This groups all events for a single span together on disk for fast materialization reads.

### 3.2. Read-Optimized Tables (`read_nodes`, `read_edges`)
These are optimized for **read performance** (Visualization).
- **Engine:** `ReplacingMergeTree(materialized_at_ms)`.
- **Purpose:** Instead of querying 2+ rows to find a span's duration, the UI queries **one** row in `read_nodes`.
- **Denormalized Fields:**
    - `flow_order`: Pre-computed sequence number.
    - `clock_skew_ms`: How much the event was shifted to stay causal.
    - `data`: Merged `Map(String, String)` of all attributes from start/end events.

### 3.3. Trace Summaries (`trace_summaries`)
The "Executive View" of a trace.
- **Purpose:** Power the "Trace List" page.
- **Contents:** Total node/edge counts, min/max importance, and **Diagnostics**.
- **Diagnostics:** Pre-calculated counts of errors (cycles, orphans, negative durations).

## 4. Why This Works (The Performance Edge)

### No Joins at Query Time
Traditional systems do: `SELECT * FROM spans JOIN edges ON ...`.
Topo-Tracer does: `SELECT * FROM read_nodes WHERE trace_id = '...'`.
Because we denormalized during materialization, the DB just streams a contiguous block of memory from disk.

### Parallel Sharding
As implemented in the `ReadOptimisedAggregator`, we process unique `trace_id`s in parallel. Since Kafka keys by `trace_id`, events for the same trace are sequential (no race conditions), but different traces can be rebuilt on different CPU cores simultaneously.

### The "ReplacingMergeTree" Strategy
ClickHouse background merges automatically collapse old versions of materialized rows. This keeps the storage footprint lean without requiring a manual `DELETE` or `UPDATE` (which are expensive in ClickHouse).

## 5. Summary Flow
1.  **SDK** → `POST /ingest` (Hono)
2.  **Hono** → **Postgres Outbox** (Atomicity)
3.  **Relay** → **Kafka** (`log.telemetry.received`)
4.  **Ingest Worker** → Writes to **ClickHouse Raw Tables** → **Kafka** (`log.trace.ingested`)
5.  **Aggregator Worker** → Runs **Materialization Logic** → Writes to **ClickHouse Read Tables**
6.  **Frontend** → Queries **Read Tables** (Lightning fast)
