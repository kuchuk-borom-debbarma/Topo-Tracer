# The Materialization Engine

Because Topo-Tracer stores telemetry as an append-only stream of fine-grained lifecycle events (`node.started`, `node.ended`), querying this stream directly for UI rendering would be far too slow. 

Instead, the backend features an asynchronous materialization engine that rebuilds "Read Models" (the current state of a node or edge) in the background.

## 1. Consumer (`LogIngestConsumer`)

When the Outbox Relay publishes `trace.events.ingested` onto the Event Bus, it is consumed by the `LogIngestConsumer`.

The consumer acts as a debouncer. If a trace receives 10,000 events in a few seconds, we do not want to rebuild the read model 10,000 times. The consumer batches dirty `traceId`s and passes them to the `ReadOptimisedAggregator`.

## 2. Aggregator (`ReadOptimisedAggregator`)

The aggregator orchestrates the rebuild. For a given `traceId`:
1.  It fetches all raw events for that trace from `LogWriteRepoClickHouse`.
2.  It feeds the raw events into the `TraceReadModelMaterializer`.

## 3. Materializer (`TraceReadModelMaterializer`)

The materializer is a pure function-like projector. It replays the event stream chronologically and computes:
-   **State**: Resolves whether a node/edge is `open` or `ended`.
-   **Durations**: Computes `endedAtMs - startedAtMs`.
-   **Flow Order**: A topological integer that orders nodes causally based on explicit edges. Nodes with no inbound edges fall back to being ordered by start time.
-   **Diagnostics**: Flags anomalies like `missingStart`, `missingEnd`, `negativeDuration`, or `orphanEdge`.

The output is an array of `ReadNodeRow` and `ReadEdgeRow` objects, stamped with a `materialized_at_ms` timestamp.

## 4. Storage (`LogReadRepoClickHouse`)

The newly computed read models are saved into `topo_tracer.node_read_nodes` and `topo_tracer.node_read_edges` in ClickHouse.

These tables use the `ReplacingMergeTree(materialized_at_ms)` engine. Because ClickHouse does not do synchronous updates, this engine ensures that background merges keep only the latest version of a node or edge, while queries use `argMax(..., materialized_at_ms)` to collapse duplicates at read time.
