# 2.3. Graph Projection

Graph projection turns a materialized trace into the bounded response consumed by the UI. It does not rebuild trace state; it shapes the latest read model into a visible window.

Main files:

- `hono-server/src/services/log/internal/service-impl/LogServiceImpl.ts`
- `hono-server/src/services/log/internal/projection/LogFlowProjector.ts`
- `hono-server/src/services/log/internal/util/CursorCodec.ts`
- `frontend/src/ui/TraceDetailPage.tsx`

## Endpoint

```txt
GET /api/v1/traces/:traceId/flow?threshold=2&limit=160&cursor=...
```

Validation in `src/index.ts` enforces:

- `threshold >= 0`;
- `1 <= limit <= 1000`;
- unauthenticated requests return `401`;
- stale or missing traces return errors through the service path.

## Service Read Path

`LogServiceImpl.projectTraceFlow`:

1. clamps the requested limit to a maximum of 1000;
2. loads the latest trace summary;
3. rejects hidden backend self-traces;
4. decodes the cursor if present;
5. rejects cursors whose `materializedAt` does not match the latest summary;
6. loads a bounded page of projection nodes by `flowOrder`;
7. loads bounded edges touching those loaded nodes;
8. calls `LogFlowProjector.project`;
9. fills paging metadata and returns the result.

The node read intentionally loads all nodes in the requested flow-order window, not only currently visible nodes. That gives the projector enough information to collapse hidden runs into ghost nodes.

## Importance Semantics

Visibility is threshold-based:

```txt
visible = node.importanceLevel <= selectedThreshold
```

Lower numbers are more important. A threshold of `0` shows only the highest-priority nodes. Raising the threshold reveals more detail.

Trace-level `importanceLabels` are metadata for humans and UI labels; they do not change filtering behavior.

## Ghost Nodes

Hidden nodes are not dropped. `LogFlowProjector` scans nodes by `flowOrder` and collapses contiguous hidden runs into a projected ghost node.

A ghost node contains:

- hidden node count;
- hidden internal edge count;
- node type counts;
- min and max hidden importance levels;
- start and end bounds;
- `flowOrderStart` and `flowOrderEnd`.

This preserves the shape of the trace without forcing the UI to render every detail node.

## Edge Snapping and Aggregation

After nodes are projected:

- normal nodes map to themselves;
- hidden nodes map to their ghost id;
- edges whose projected source and target are the same ghost increment `hiddenEdgeCount`;
- edges that cross projected nodes are aggregated by `fromNodeId`, `toNodeId`, and `edgeType`;
- edges that reference nodes outside the loaded page increment `omittedEdgeCount`.

The response edge has `edgeCount` so the UI can represent multiple collapsed parallel edges.

## Paging and Cursors

Cursor state stores:

- `offset`;
- `materializedAt`.

The `materializedAt` check is important. If a trace is re-materialized between page reads, the old cursor may point to a different flow-order window. The service rejects that stale cursor and asks the client to refresh.

Response metadata includes:

- `nextCursor`;
- `previousCursor`;
- `hasAfter`;
- `hasBefore`;
- `totalNodeCount`;
- `fromFlowOrder`;
- `toFlowOrder`.

## Hard Caps

Current caps:

- route `limit`: maximum 1000 nodes;
- service projection limit: maximum 1000 nodes;
- repository edge cap: 2000 edges;
- trace list limit: maximum 100 summaries.

These caps are part of the product design. The UI should page and threshold rather than ask the backend for the entire graph.
