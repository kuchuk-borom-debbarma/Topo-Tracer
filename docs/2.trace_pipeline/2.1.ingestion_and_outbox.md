# Trace Ingestion and Outbox Relay

Trace ingestion is the highest-throughput path in Topo-Tracer. It must be fast, resilient to failure, and guarantee that ingested events eventually trigger the materialization engine.

## 1. Request Handling (`LogServiceImpl.ingestNodesNEdges`)

When the SDK sends a batch of telemetry events to `POST /api/v1/ingest`, the request hits the Hono controller and is passed to `LogServiceImpl`.

1.  **Validation:** The service checks the payload for malformed IDs or impossible timestamps.
2.  **Raw Append:** The service inserts the raw events into ClickHouse via `LogWriteRepoClickHouse`. This is an append-only operation to `topo_tracer.node_trace_events`.
3.  **Outbox Record:** Crucially, to trigger downstream processing without blocking or losing data if the Event Bus goes down, the service writes an event to the Postgres Outbox (`outboxStore`).

Because Postgres provides strong transactional guarantees, the outbox record ensures that the system *remembers* to process this trace.

## 2. The Outbox Relay Daemon

Instead of publishing to the Event Bus inline during the HTTP request, Topo-Tracer uses a background daemon (`OutboxRelay`) that runs continuously in the Node/Bun process.

1.  **Polling:** The relay polls the Postgres `outbox_events` table for unpublished events.
2.  **Publishing:** It takes batches of events (e.g., `trace.events.ingested`) and publishes them to the `IEventBus` (which may be Kafka or an in-memory development bus).
3.  **Committing:** Once the Event Bus acknowledges receipt, the Outbox Relay deletes or marks the outbox rows as processed in Postgres.

## Why this pattern?
If ClickHouse succeeds but the Event Bus is unreachable, the outbox record remains safely in Postgres. When the Event Bus recovers, the daemon automatically resumes publishing. This achieves **at-least-once delivery** to the materialization engine without slowing down the HTTP ingestion response time.
