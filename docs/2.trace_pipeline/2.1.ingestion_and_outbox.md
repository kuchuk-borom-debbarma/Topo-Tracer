# 2.1. Ingestion and Outbox

Ingestion is the write side of Topo-Tracer. Its job is to accept telemetry quickly, preserve raw facts, and trigger downstream read-model work without coupling the HTTP request to graph rebuilding.

## Ingested Event Types

The backend accepts five arrays in one batch:

- `traceStarts`: trace metadata such as name and importance labels.
- `nodeStarts`: node identity, type, start timestamp, attributes, display name, and importance.
- `nodeEnds`: node end timestamp and end message.
- `edgeStarts`: explicit graph link from `fromNodeId` to `toNodeId`.
- `edgeEnds`: edge end timestamp.

These types are declared in `hono-server/src/services/log/api/types.ts` and mirrored by the Node SDK under `sdks/node-js/src/types.ts`.

## HTTP Boundary

`POST /api/v1/ingest` is implemented in `hono-server/src/index.ts`.

The route:

1. authenticates with `jwtAuthMiddleware`;
2. reads the JSON body;
3. checks that `nodeStarts`, `edgeStarts`, `nodeEnds`, and `edgeEnds` are present;
4. passes the resolved `userId` and event arrays to `logService.ingestNodesNEdges`;
5. returns `{ "success": true }` once the event has been published to the event bus.

The HTTP route does not write ClickHouse rows directly.

## Service Publish Step

`LogServiceImpl.ingestNodesNEdges` validates the batch and publishes one `log.telemetry.received` event.

Important behavior:

- empty batches are ignored;
- every `edgeStart` must have a non-empty `fromNodeId` and `toNodeId`;
- the idempotency id is derived from user id and event identities;
- the event key is `userId`;
- the event data contains the original normalized batch.

The service publishes through `IEventBus`, so the same service code works with `DevEventBus` or `KafkaEventBus`.

## Raw Persistence Step

`LogIngestConsumer` subscribes to `log.telemetry.received`.

For each received batch it:

1. normalizes missing arrays to empty arrays;
2. writes raw rows through `LogWriteRepoClickHouse`;
3. computes the unique affected `traceId` values;
4. publishes one `log.trace.ingested` event per affected trace.

Raw rows are appended to:

- `trace_events`;
- `node_events`;
- `edge_events`.

These tables use ClickHouse `MergeTree` engines and are treated as append-only event logs.

## Rebuild Signal Step

`log.trace.ingested` is the durable instruction that a trace has new raw events and should be materialized.

The event payload is small:

```json
{
  "userId": "user-id",
  "traceId": "trace-id"
}
```

The event key is `traceId`, which gives Kafka a natural ordering lane for one trace. The materializer still handles duplicate work because the bus is treated as at-least-once.

## Outbox Infrastructure

The event bus supports an outbox path when callers publish with a transaction object:

- `DevEventBus.publish(events, { tx })` stores rows through `IOutboxStore`.
- `KafkaEventBus.publish(events, { tx })` does the same.
- `OutboxRelay` polls `outbox_events`, publishes pending work with `bypassOutbox: true`, and marks rows sent.

Current auth signup uses this transactional outbox path when it inserts the pending user and publishes `auth.signup.started` inside one Postgres transaction.

Current telemetry ingestion publishes `log.telemetry.received` without a transaction object, so the event bus delivers it directly. Raw ClickHouse insertion happens in `LogIngestConsumer`, not in the request transaction.

## Delivery Guarantees

The pipeline is designed around at-least-once delivery:

- publishers provide stable `idempotencyId` values;
- bus implementations filter duplicate deliveries per `consumerName`;
- consumers mark idempotency only after handlers complete;
- materialization is idempotent because latest state is folded by ids and saved as versioned read rows.

## Failure Behavior

- If route validation fails, the request returns `400`.
- If auth fails, the request returns `401` or `403`.
- If publishing `log.telemetry.received` fails, ingestion returns `500`.
- If `LogIngestConsumer` fails after delivery, the event is not marked processed and can be retried.
- If materialization fails, the checkpoint is not advanced and later delivery can retry from the previous checkpoint.
