# 9.1. ClickHouse Checkpoint Lookback Journal

This entry documents the investigation, root cause, and fix for the trace lineage fragmentation bug in the Topo-Tracer ingestion pipeline.

---

## 1. The Bug: Disconnected Trace Graphs
When visualizing traces from the `ingestNote` endpoint, the trace graphs displayed in the UI were fragmented. High-level spans (like `HTTP POST` and `AIController`) were rendered, but the inner logic spans and sequential logs (such as database calls, candidate extraction logs, and preprocessing status) were completely missing from `read_nodes` and `read_edges`.

---

## 2. Root Cause: Checkpoint Pagination Overlap
The Hono backend uses an incremental materializer to process new events in batches. A checkpoint of the last processed event is recorded in the `materialization_checkpoints` table:
* `lastNodeEventTime` (timestamp of the last node event)
* `lastNodeEventId` (UUID of the last node event)
* `lastNodeEventType` (0 = start, 1 = end)

To fetch events ingested after this checkpoint, the repository queried:
```sql
SELECT * FROM node_events
WHERE user_id = {userId:String} AND trace_id = {traceId:String}
AND tuple(if(event_type = 0, assumeNotNull(started_at_ms), assumeNotNull(ended_at_ms)), id, event_type) > 
    tuple({lastNodeEventTime:UInt64}, {lastNodeEventId:String}, {lastNodeEventType:UInt8})
ORDER BY if(event_type = 0, assumeNotNull(started_at_ms), assumeNotNull(ended_at_ms)), id, event_type
```

### The Lexicographical Skip Condition
ClickHouse compares tuples element-by-element from left to right:
`tuple(timeA, idA, typeA) > tuple(timeB, idB, typeB)`
If `timeA == timeB`, it evaluates `idA > idB` (lexicographical string comparison).

If the SDK flushes events in separate batches (e.g. because of the 5-second periodic flush timer), events for the same trace will be processed across different materialization cycles:
1. **Cycle 1:** Processes the first batch, ending at timestamp `1781480976387`. The last event has ID `5923d5ed...`. The checkpoint is updated to `(1781480976387, '5923d5ed...', 1)`.
2. **Cycle 2:** A new batch is ingested. It contains an event `IngestNote ...` (ID `1e14e93a...`) that also occurred at the exact same millisecond `1781480976387`.
3. **The Query:** Evaluates:
   ```sql
   tuple(1781480976387, '1e14e93a...', 0) > tuple(1781480976387, '5923d5ed...', 1)
   ```
4. **The Failure:** Since `1781480976387 == 1781480976387`, it compares the IDs. Because `'1e14e93a...'` is lexicographically **smaller** than `'5923d5ed...'`, the inequality returns `false`.
5. **Result:** The `IngestNote ...` event is **permanently skipped and lost**.

This bug also affected late-arriving events that were flushed out of chronological order due to parallel execution or network routing. Any event with a timestamp older than the checkpoint's timestamp was completely ignored.

---

## 3. The Solution: Checkpoint Lookback Window
Since updating the read models is fully idempotent, the safest and most robust solution is to introduce a **lookback window** (e.g., 10 seconds / 10,000 ms) when querying ClickHouse.

Instead of querying strictly after the checkpoint, we query starting from 10 seconds before the checkpoint's timestamp, resetting the tie-breaker ID and EventType:

### Example of Checkpoint Adjustment
Given the current checkpoint:
```json
{
  "lastNodeEventTime": 1781481224523,
  "lastNodeEventId": "841ebfff-8f9c-4839-a93e-182a132acb1a",
  "lastNodeEventType": 1
}
```

The repository adjusts it prior to building the query parameters:
```typescript
const lookbackMs = 10000;
const nodeCheckpoint = params.checkpoint ? {
  lastNodeEventTime: Math.max(0, params.checkpoint.lastNodeEventTime - lookbackMs),
  lastNodeEventId: "",
  lastNodeEventType: 0,
} : {
  lastNodeEventTime: 0,
  lastNodeEventId: "",
  lastNodeEventType: 0,
};
```

This translates the query predicate to:
```sql
tuple(time, id, type) > tuple(1781481214523, "", 0)
```
which behaves exactly as:
```sql
time >= 1781481214523
```

### Why this is safe
1. **Idempotency:** The materializer folds nodes and edges into `Map` structures keyed by `id`. Reprocessing events already stored in the read model simply overwrites them with the same data.
2. **Performance:** The ClickHouse query is strictly scoped by `trace_id` and `user_id`, meaning the database only scans and streams a tiny subset of rows (usually less than a hundred events) even with the lookback window.
